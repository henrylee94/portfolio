Workflow 主干——每天 8.4M 条审计记录

保证钱永远不会被错误地动两次的系统。

每天 8.4M 条记录 峰值 ~1k/s 100+ 次 SDK 发布 2 分钟自助接入 585 个测试,跑真实容器 236 种请求类型

背景

平台上每一个动钱的操作——提现、充值、奖励、配置变更——都必须走同一条带审计的 create → review → approve → execute 生命周期。旧实现和单体紧紧耦在一起,每个接入方都要自己重新实现一遍接线。

约束

这是整个平台唯一的信任锚点:生产环境每天有 ~8.4M 条记录从这里过,一次重复执行,就意味着真实的钱被动了两次。

做法

把它重建成由 Temporal 编排的 gRPC 微服务:一条通用的持久化 workflow 跑所有请求类型,业务扩展点做成 child-workflow 插槽。幂等是结构性的——workflow ID 兼任创建 key,所以被 retry 的步骤只会拿回已存在的记录,而不是创建第二条。接入变成对外发布的 SDK:任何服务一个 import 就能注册 executor;新增一种请求类型是一张自助 CI 表单,而不是排在我桌上的队。100+ 次 SDK 发布,SemVer 纪律。

结果

生产环境每天 8.4M 条记录。重复执行由构造防住,而不是事后去抓。新类型接入从一个 30–60 分钟、依赖特定人的流程,变成了两分钟自助。

实战故事

没扣费的那次兑换

最先发现的是客服,不是任何一块 dashboard:一条奖励兑换流程在发放价值,却没有扣减余额。workflow 里的预扣费步骤被跳过了——而且是静默跳过。

先 rollback,再排查。线索指向一份共享的契约文件:它在某个分支上悄悄落后了,一次例行的包升级激活了这份过期副本。测试从没抓到它,因为改动来自另一类项目——两个各自安全的变更,组合成一个不安全的系统,是最经典的盲区。

修复是系统性的,不是局部的:预扣费步骤没跑,workflow 就大声失败;每次发版都对每一条动钱的路径做回归测试,没有例外;事故报告向上、向外同步——全队现在都认识这种失败形态。

还有一样东西从这次事故里长了出来:我们风控告警 pipeline 的第一版——每个兑换事件都发布到事件总线上,在聊天里告警风控团队。先做检测,自动化后置。事故最便宜的归宿,是把它编译进系统。

来得太早的 signal

我们的 workflow 把人工决策和第三方确认都当作 signal 接收——有一天,一个回调在我们自己的状态还没持久化完成之前就到了。workflow 收到了一个 signal,而它指向的记录在 workflow 看来还不存在;更糟的是,replay 时这个交错顺序并不确定。修复是结构性的:signal 会被缓冲,直到它指向的状态落地为止;并且 workflow 代码的每一次行为变更都带着版本化的 patch 标记上线——历史跑到一半的 workflow 会 replay 它们启动时的那套逻辑,新的才走新路径。在持久化 workflow 里,一次代码 deploy,就是你正在运行的历史必须活过去的一个事件。

必须拿回同一条记录的 retry

引擎会 retry 失败的步骤——这本来就是它存在的意义。但在资金路径上,一次被 retry 的 create 就是一条潜在的重复记录,而一条重复记录就是钱被动了两次。所以创建不是靠事后补一个检查来守护,而是由构造守护:workflow ID 兼任记录的创建 key,每次 create 都先查再写——retry 找到第一次尝试创建的那条记录,把它返回。更新带状态守卫,非法的状态迁移直接抛错而不是写入。当失败模式是金钱时,幂等不能是尽力而为;它必须是不可能做错。

四个环境,一个引擎

几个环境共用一个 workflow 引擎集群,worker 开始互相偷任务——一个开发者本地的 worker 可能悄悄捡走并搞挂一条属于共享测试通道的任务。修复靠的是命名纪律,不是更多硬件:task-queue 名称按环境解析——生产用固定名,本地运行带上开发者身份后缀,测试通道带环境后缀。很无聊,但它终结了一整类「我机器上是好的,你的测试却挂了」的幽灵问题。共享基础设施先需要 namespacing,然后才需要容量。
