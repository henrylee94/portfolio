02没有走的路

没有走的路

我现在跑的每一套架构,都替换掉了一套我亲手维护过的。这里是落选者,和它们输掉的原因。

Cron 加扫描,还是 durable workflow

旧平台跑着 22+ 个 cron job,其中一个每分钟重扫一遍卡住的审批。这套东西我维护了六年——所以 durable workflow 一来,不需要谁写博客来说服我:retry 应该住在引擎里,而不是扫描器里。什么会让我改主意:一个所有任务都真正 fire-and-forget 的场景。

把数据库调得更狠,还是加一层缓存

Redis 不是第一个答案——先来的是几个月的 index 功课、aggregation 重写、删掉死 index。等数据库已经诚实了,缓存才配拿到自己的位置。什么会让我改主意:没有;这个顺序本身就是原则。

输掉的 benchmark,后来赢了

一个 OLAP 引擎在我们的 JSON-path 查询上败下阵来——慢 3 到 15 倍。我们没有接受判决,而是把 schema 拍平、移除弱点,判决就变了。Benchmark 是架构的输入,不是它的法官。

本地模型,还是分层 tiered router

在家里我什么都想用本地模型——免费、私有、属于我自己。CPU-only 推理在多轮 tool calling 上输了一个数量级,于是 tiered router 顶了上来。干脆地输,好过纸面上赢。
