01AI 工程

AI 工程

用好 AI 是一门工程学科,不是一个习惯。我团队的 AI 交付体系是我搭的——pipeline、门禁、context、review 角色——它交付的是真实的功能。

10 个 pipeline 阶段 24 个 skills · 8.7k 行 16 个功能经它上线 32 份按服务的 context 文件 4 个按角色分工的 AI reviewer 功能初稿:数天 → 数小时

02失败档案

失败档案

你读到的每一个 AI 成功故事,都缺了这一节。这是我们这套体系里出过的错——纪律正是从失败里长出来的。

把生产搞坏的那次修复

agent 找到了一个真 bug——在生产上无害地躺了好几年——然后把它修了。这一修,改掉了其他代码早已悄悄依赖的行为,东西就炸了。教训:agent 评估的是代码,它不知道你系统的历史。一个无害的 bug,在证明之前都要当它是承重的。

小需求,大答案

需求真的很小——给一条现有规则多加一个条件,代码库本来就是为这种改动准备的。agent 没抓住这层意图,把周边的流程重建了一遍:几百行新代码,技术上能跑,review 负担翻倍。教训:最小的正确改动,来自知道代码库本来想怎么做——这个判断仍然在我们手上。

正确,但慢

要求它保证准确,agent 就一行一行地更新数据库——正确,但在我们的量级上是一场 IO 风暴。批量处理是显而易见的答案,可只对为 IO 付过学费的人显而易见。教训:性能直觉不随模型出厂,它来自伤疤。

三件事背后是同一个模式:当一个人还分不出「好答案」和「像样的答案」时,伤害就发生了——junior 在错误的问题上烧 token,merge 自己解释不了的改动。所以规则不是「少用 AI」,而是:先有判断力,再谈委派。你得先能评判问题和逻辑,才能把打字外包出去。

04与 agent 协作

与 agent 协作的原则

  • AI 提议,人来批准。每一个不可逆的动作——commit、publish、部署、migration——都在一道明确的人工门禁后面。任何 agent 都不许 commit。
  • AI 压缩的是从零到八十的那段路。两天的活变成两小时的初稿。最后那百分之二十仍然归我们,真正的本事是判断力:抓出它做错的地方。
  • AI 放大你的代码文化。喂进去什么,长出来什么——在有纪律的代码库里养大的 agent,交付纪律;在烂摊子里养大的,只会更快地交付烂摊子。
  • Context 胜过 prompt。agent 接到任务时就该已经认识代码库——路由清单、按服务的 context 文件、持久记忆——而不是靠一句措辞巧妙的请求。
  • 事故要编译成规矩。一次生产教训变成写进 agent harness 的规则——一道 agent 跳不过的就绪门禁,同一个错误就不可能上线第二次。

05规矩

我的 agent 要守的规矩

我的两个 agent——公司那个和家里那个——跑在同一套长期规则下。每一条都是踩过坑才写下来的。

  1. 规则 0:不许猜。先验证再开口;验证不了,就直说验证不了。
  2. 我的话就是字面意思。不要解读,不要替我补完想法——问。
  3. 只执行被要求的事。「顺手把这个也……」正是事故的起点。
  4. 每次写入后都要给证据。「做完了」是一句主张,文件路径才是证据。
  5. 讨论不等于许可。聊到要跑某个东西,不等于批准去跑。
  6. 「跑过了」不等于「跑对了」。每次执行后都要验证。

慢不要紧,错不行。

06养 agent

养一个 agent

实验室那页展示的是我家里那个 agent 会做什么。这一页讲它是怎么长成这样的——和我在公司调教 agent 用的是同一套方法,跑了两遍,收敛出同一部宪法。

先立宪法,再派活

agent 出生的第二天,我就给它装上了 SOUL.md——它的身份文件:解题协议、命名规则、行为法——在它干任何正经活之前。规矩这种东西,事后补装是装不牢的。

小活先行

新 agent,只 delegate 小范围的活。我盯着它在哪儿卡壳、在哪儿越界,再逐步放宽。这和我带 junior 工程师用的,是同一把校准过难度的阶梯。

失败即立法,当天生效

一个 session 里,它三次没等确认就执行——当天,「讨论不等于许可」就成了永久法条。六条长期规则,每一条背后都有一次事故。

写入要出示证据

每次纠正之后,agent 必须把存进去的条目亮出来——ID 和内容。一句「记住了」,不能证明学习真的发生过。

记忆有经济学

每一条常驻加载的事实都要永远付 token 成本,所以 MEMORY.md——常驻加载的那一层——设了「7 天准入测试」:一周后还真、还有用,才准进。事实按分量分流——行为法每一轮都加载,Mnemosyne 存放长尾、按需召回,聊天记录可搜索,但不占道。

连 harness 一起调,不只调 prompt

贵的模型被过度用在简单执行上,于是降级为只值 deep-reasoning 的班。一次更新抹掉过一回 config——当天就加上了更新前备份。每晚一份记忆 digest,每周一轮整合,每月一次自查:agent 按日程维护它自己。

两个 agent,两套独立记忆,收敛出同一部宪法——这时候你才知道,这是方法,不是习惯。

07约束

每个工具都在回答一个约束

这些工具没有一样是因为"酷"才存在的。大语言模型有几条绕不开的硬约束——我的每一件工具,都是对其中一条的直接回答。

上下文是有限的

模型装不下所有东西,所以"相关"比"多"重要:一份覆盖 17 个 repo 的路由清单决定任务归哪里,30+ 份 per-service context 文件让 agent 一开工就认识代码库——而不是每个 session 重新读一遍。

记忆不会自己留下

agent 最大的残障:session 一结束,知识就没了。我的答案是分层——身份文件放"法律",常驻事实文件设 7 天准入测试,向量库(Mnemosyne,3,700+ 条)接长尾,Obsidian 做人工沉淀层。事实按权重路由;重要的东西绝不只活在一个脑袋里。

token 不是免费的

agent 跑的每条 dev 命令都在烧上下文。一层过滤代理(RTK)在模型看到之前把噪音剥掉——实测省下 75M+ tokens。不需要判断的定时任务干脆不用 LLM 跑。在家里,分层路由把日常活派给便宜模型,重推理才请重型选手。同一个原则贯穿所有尺度:为思考付钱,不为打字付钱。

像样 ≠ 正确

模型产出的是"看起来对"的答案。所以任何东西不过人工闸门不许 merge,每次记忆写入都要出示收据,上面那份失败档案也因此存在。信任是预算——花在验证上,不要花在侥幸上。

好流程不该重复自己

做过两遍的事就该变成有版本的 skill。有些是从社区拿来再改造的——署名,不冒领;有些是从零自建的:交付管线、角色评审、repo 路由。检验标准永远是同一条:一个新同事——不管是人还是 agent——第一天照着做,能不能做对?

工具会换,约束不会。为约束而建。
