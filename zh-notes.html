01实战笔记

实战笔记

来自生产环境的短课——那种只有凌晨两点才学得到的。

+10 亿的偏移

我们要在用户在线的情况下,把 ID 生成从数据库挪到 Redis。没有维护窗口——所以我们没去协调一次 cutover,而是让冲突从根上不可能发生:新的号段直接从旧计数器往上加 10 亿开始。交易在盯着的时候,无聊胜过聪明。

cluster 说不的那天

用包在 MULTI 里的原子自增修好了一个读-改-写竞态。结果生产环境告诉我们:我们的 Redis cluster 不支持 MULTI。当天就降级成纯原子操作。一个修复,要在真实拓扑里活下来才算修完。

一个数字,一个 owner

一个阈值同时活在两个系统里,总有一天会悄悄跟自己打架。我们删掉了生产者那边的过滤,让这个数字只存在于一个地方。debug 本质上是图搜索——边越少,赢面越大。

幂等是契约,不是补丁

在交易路径上,我们假设每一步都会被 retry,因为总有一天真的会。每一次写入都可以安全地重放;workflow ID 同时就是创建的 key。为 retry 而设计,凌晨两点的报警就永远不会来。

AI 起草,人来把关

AI 把 0 到 80 那一段压缩掉了;最后的 20% 依然是我们的。真正值钱的能力是判断力——在 merge 之前抓住模型写错的地方。而且它会放大它所处的代码文化:烂进,烂出。

交接也是一个 feature

我从零建了一个客服平台,然后把它交了出去——交了两次。一个系统如果离了你就活不下去,它就还没做完。文档、runbook、自助表单不是杂务;那是你离开的方式。

一行裸的 Server::serve(),后来成了制度

有个服务上线时没有处理信号——每次 deploy 它都死在请求进行到一半的时候。这个修复后来变成了写进我们 agent harness 的规则:graceful shutdown 是一道 readiness 门槛,任何动到服务入口、却没走 drain 清单的改动都会被拦下。事故最便宜的归宿,是被编译成制度。

一个 lead 每天看什么

每天早上站会之前:消息队列的积压深度、workflow retry 的次数、慢查询日志、热点服务的 CPU 与内存——还有内存泄漏在惊动任何人之前的形状。看板在 Grafana;基于日志的告警与线上取证在阿里云 SLS;底层是 OpenTelemetry 链路与 Prometheus 指标。不是因为盯着看让人安心——是因为每一条曲线背后都有一个故事,我想在用户之前先听到它。

修复脚本也是生产代码

每当交易或数据需要回填,总会有人在半夜写一个「一次性」脚本。我们的从来不是一次性的:分 batch、可断点续跑、幂等、先 dry-run——因为一个跑到一半失败的修复脚本,就是第二起事故。这种脚本我写过的多到数不清;让它们保持无聊的,正是纪律。

先索引,再缓存

在平台配得上它的 Redis 层之前,好几个月的功夫都花在数据库本身:covering index 对着真实的查询形状建,aggregation pipeline 一段一段地重写,过去靠 regex 扫描慢慢爬的地方换成了全文检索。最快的缓存,是那条已经不需要缓存的查询。
