01案例 02 / 07

平台扩容:5 万 → 30 万+ 用户

平台涨了六倍;我的工作是让所有人毫无察觉。

6 倍增长,线上完成 cutover 偏移 +1e9 亚秒级推送 查询窗口 1 天 → 30 天

背景

四年里平台涨了 6 倍,增长不断把旧机制的极限找出来——数据库里的一份 counter 文档把所有 ID 生成串行化;通知 fan-out 在高负载下卡死。不止一次,故障是先在生产上发生的。

约束

没有重写的预算,也没有停机的余地——每个修复都必须塞进每周发版的节奏里,在线上落地,用户还在线。

做法

把每个瓶颈都追到它的机制层面,然后换掉机制,而不是治症状:ID 生成从数据库 counter 换成原子的 Redis INCR——cutover 带 +10 亿的偏移,新旧区间永远不可能相撞;通知 fan-out 迁到 Redis pub/sub(上线过,因生产意外 revert 过,加固后重新落地);阻塞式的全量扫描换成基于 cursor 的 SSCAN;一个泄漏的内存配置缓存重建成 15 分钟刷新一轮。

结果

平台稳定支撑 300k+ DAU,旧的失败模式不再出现。这次引入的模式(cache-aside、原子计数、pub/sub fan-out)成了之后新模块的团队标准。

实战故事

调优的那几个月

这个平台在配得上它的 Redis 层之前,得先配得上它的数据库。报表层比任何 BI 团队都出现得早——有一段增长期把它明晃晃地压垮了:运营报表干脆不再返回,最忙的那个 collection 上,一条查询连一天的记录都取不回来就超时。

之后的几个月毫无光环:像读报纸一样读 slow-query 日志,给大家真正在跑的查询形状配上 covering index(再删掉没人用的索引——在写多的 collection 上,一个没用的索引就是纯税),把 aggregation pipeline 一段一段重建,把文本搜索从 regex 扫描搬到真正的搜索索引上。

30 天窗口的报表,在那些曾经连 1 天都超时的 collection 上轻松跑回来了。这些工作从没上过发版公告——但之后上线的每个功能都踩在它上面。

先测量,再争论

推送通知到得晚,屋里的直觉是去调 WebSocket 层——它是看得见的那个组件。动手之前我先量了时间到底花在哪,结果成本根本不在 socket 上:是它背后的 fan-out 循环,从一个地方向每条连接投递。我们把 fan-out 迁到 pub/sub,让每个实例投递自己的连接,推送就进了亚秒级。最吵的组件很少是瓶颈;profile 说了算,直觉不算。

回来更强的那次 revert

fan-out 迁移上线了——生产给了我们一个意外,于是 revert。第二次落地长得完全不同:不再是一个大改动,而是一串小的、各自可 revert 的步骤,每一步都被第一次尝试教到的东西加固过。设计没有错;错的是交付方式。在一个跑着真实交易的线上平台,变更的单位必须等于 rollback 的单位——这条规则从这场仗里来,之后再没离开过。

领先 10 亿个 ID 的起跑线

ID 生成必须从数据库 counter 迁到 Redis 的原子自增——用户在线,没有维护窗口。与其在两个活着的生成器之间编排一次精准的 cutover,我们直接让碰撞不可能发生:新区间从旧 counter 之上 10 亿的位置起步,两边可以同时跑,永远碰不上。rollback 同样无聊——指回旧 counter 就行。当交易在盯着的时候,聪明的方案,是那个没有任何东西可以再出错的方案。
