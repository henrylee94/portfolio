Transaction Backbone

Rebuilt a legacy transaction flow into a durable workflow service with safe retries and self-service integration.

Role

Module owner — designed the service, led the migration, run it in production today.

Scale

Millions of transactions a day, peaks around a thousand per second; 200+ request types on one base workflow.

Stack

NestJS · gRPC · Temporal · MongoDB · published SDK with semantic versioning.

Outcome

Duplicate execution prevented by construction; new-type onboarding cut from 30–60 minutes to ~2; 100+ SDK releases.

Problem

Every financial transaction on the platform — withdrawal, top-up, loyalty reward, configuration change — must pass one audited create → review → approve → execute lifecycle. The legacy implementation was tightly coupled inside a monolith: every consumer re-implemented its own wiring, retries lived in cron-driven scanners, and adding a new request type was a 30-to-60-minute, person-dependent process that queued on the core team.

Traffic growth made this untenable — the flow was heading past what the old architecture could hold, and any duplicate execution on this path means a transaction is processed twice.

Constraints

  • Production traffic could not stop: the platform's single point of trust, live throughout the migration.
  • Financial correctness — a retried create must never produce a second record.
  • Backward compatibility — legacy consumers stayed on the old endpoints until each flow cut over, with endpoint-level fallback.
  • Auditability — every state transition reviewed, logged, and traceable.
  • Cross-team integration — downstream services on different schedules had to adopt at their own pace.

Role and responsibility

I own this module. I designed the service, wrote the initial production version, planned the migration path from the monolith, and coordinated the gRPC contracts consuming teams integrate against. The SDK and its release discipline are mine; the request-type patterns other services copy started here. Implementation beyond the initial version was shared with my team.

Architecture

A gRPC microservice orchestrated by Temporal: one generic durable workflow runs every request type, with business extension points as child-workflow slots. Idempotency is structural — the workflow ID doubles as the creation key, so a retried step returns the existing record instead of creating twice. Integration is a published SDK: any service registers an executor with one import.

Lifecycle

create → review → approve → execute, one audited path for every transaction type. Human review sits in the middle; the engine owns retries and timeouts.

Walkthrough — what each state does

First request: the workflow ID is derived from the request, the record is created, the lifecycle starts. Retry: the same derivation finds the existing record and returns it — no second write. Worker failure: the engine replays the workflow history on another worker; completed steps are not re-executed. Existing record return: the caller receives the same transaction ID either way.

Key decisions

Durable workflows over cron-and-scan

Options
Keep the legacy pattern — cron jobs re-scanning for stuck approvals — or move orchestration into a durable workflow engine.
Decision
Temporal. Retries, timeouts and recovery belong in the engine, not in scanners.
Reason
I had personally operated the 22-cron-job version for years; the failure modes were known and structural.
Trade-off
A new operational surface: worker fleets, task queues, and replay-deterministic code discipline the team had to learn.

One generic workflow, not one per request type

Options
A workflow implementation per transaction type, or one base workflow with business extension slots.
Decision
One base workflow; 200+ request types run through it.
Reason
Every new type inherits idempotency, audit and retry behaviour for free — correctness is written once.
Trade-off
The base workflow is high-blast-radius code: changes ship behind version markers and get the strictest review on the team.

Cutover carried by the SDK, with endpoint-level fallback

Options
Big-bang switch, or migrate flow by flow with the SDK routing between old and new endpoints.
Decision
Framework first, then flow-by-flow cutover; every endpoint could fall back independently.
Reason
On a live financial path, each step needs its own rollback — a big-bang cutover has none.
Trade-off
Two systems ran in parallel during the migration window, which meant double observability and double support until retirement.

Failure modes and recovery

Real incidents and the mechanisms that came out of them.

The redemption that didn't charge

Customer support found it before any dashboard did: a reward-redemption flow was handing out value without deducting the balance. A pre-charge step in the workflow was being skipped — silently. Rollback first, debug second. The trail led to a shared contract file that had quietly fallen behind on one branch; a routine package upgrade activated the stale copy.

The fixes were systemic, not local: the workflow fails loudly if the pre-charge step doesn't run; every release regression-tests every transaction path; and the incident seeded the first version of our risk-alerting pipeline — every redemption event published onto the event bus, alerting the risk team in chat.

The signal that arrived too early

A third-party callback landed before our own state had finished persisting — the workflow received a signal for a record that, from its point of view, didn't exist yet. The fix was structural: signals are buffered until the state they refer to has landed, and every behavioural change ships behind a versioned patch marker, so workflows already running mid-history replay the exact logic they started with. In durable workflows, a code deploy is an event your running history has to survive.

The retry that had to return the same record

The engine retries failed steps — that's the point of it. But on a transaction path, a retried create is a potential duplicate record. So creation isn't guarded by a check bolted on afterwards; it's guarded by construction: the workflow ID doubles as the record's creation key, and every create looks up before it writes. Updates carry status guards, and illegal state transitions throw instead of writing.

Four environments, one engine

Several environments shared one workflow-engine cluster, and workers began stealing each other's tasks. The fix was a naming discipline, not more hardware: task-queue names resolve per environment — production fixed, local runs suffixed with the developer's identity, test lanes with the environment. Shared infrastructure needs namespacing before it needs capacity.

Representative code

Rewritten for publication to show the production design — not company source.

create-record.workflow.ts 
// Idempotency by construction — the workflow ID doubles as the creation key.
async function createRecord(req: CreateRequest): Promise<TxRecord> {
  const workflowId = `tx-${req.type}-${req.requestId}`;

  // Look up before write: a retried step finds the record its
  // first attempt made — and returns it. Never a duplicate.
  const existing = await repo.findByCreationKey(workflowId);
  if (existing) return existing;

  return repo.insert({ creationKey: workflowId, ...toRecord(req) });
}
base.workflow.ts 
// One generic workflow runs every request type. A third-party callback
// can arrive before our own state has landed — so signals are buffered
// until the record they refer to exists, and behaviour changes ship
// behind a version marker so in-flight histories replay deterministically.
export async function proposalBaseWorkflow(req: Request) {
  const buffered: Signal[] = [];
  setHandler(decisionSignal, (s) => buffered.push(s)); // never dropped

  const record = await createRecord(req);         // idempotent create
  await condition(() => buffered.length > 0);      // drain once state exists

  if (patched('settle-v2')) return settleV2(record, buffered);
  return settleV1(record, buffered);       // old runs keep old logic
}

Testing and validation

Integration tests run against real dependencies via Testcontainers, not mocks. The test that matters most on this path: fire the same create twice, assert one record exists and both responses carry the same transaction ID.

create-record.integration.spec.ts 
// Real database in a container — the guarantee is tested, not assumed.
it('returns the same record when the same request is sent twice', async () => {
  const req = makeCreateRequest({ type: 'withdrawal', requestId: 'r-100' });

  const first  = await service.createRecord(req);
  const second = await service.createRecord(req); // duplicate delivery

  expect(second.id).toBe(first.id);
  expect(await repo.countByCreationKey(first.creationKey)).toBe(1);
});

Trade-offs

  • Temporal brought real operational complexity — worker fleets, task-queue management, and a replay-determinism discipline every contributor has to learn before touching workflow code.
  • During migration, the legacy path and the new service ran in parallel: duplicated infrastructure, double monitoring, and a longer tail of support until the old flows retired.
  • The generic base workflow concentrates risk: a defect there touches every request type, which is why changes to it move slower than anywhere else in the module.

Outcome

  • Millions of transactions a day in production, still climbing.
  • Duplicate execution prevented by construction rather than caught after the fact.
  • New-type integration went from a 30–60-minute, person-dependent process to ~2-minute self-service.
  • 100+ SDK releases under semantic versioning; 200+ request types on one base workflow.
  • The SDK-per-domain pattern became the template later platform services copied.

What I learned

On a financial path, idempotency can't be a best effort bolted on afterwards — it has to be impossible to get wrong, which means it lives in the data model, not in a check. And durable workflows change what a deploy is: code changes become events your running histories have to survive, so versioning discipline matters as much as the business logic. Finally, the highest-leverage thing I built here wasn't the service — it was the self-service integration path that took the core team out of every consumer's critical path.
