Support System

Built from zero on a stack new to the company — AI-assisted handling with human takeover — grown to a team and handed off clean.

Role

Built it solo from zero, grew the team to two, then handed the service off — first to a maintenance team, later to the AI team.

Scale

Real-time customer conversations for the platform; AI designed to absorb roughly 70% of repetitive inquiries, humans take the rest.

Stack

NestJS · gRPC · Temporal · Pulsar · Redis · a self-hosted off-the-shelf messaging backbone as transport.

Outcome

First production service on the new stack; its patterns became the template later services copied. Runs today as standalone multi-channel middleware.

Problem

The platform needed a real-time support channel and had none. At the same time, the company had just decided on a new microservices stack — NestJS, gRPC, Temporal, Pulsar — that no one had shipped to production yet. I was handed both problems at once: build the support system, and prove the stack on the way.

This was the pilot the whole stack migration was judged on. Whatever patterns it set — good or bad — later services would inherit.

Constraints

  • Solo start — one engineer, no existing service on the stack to copy from.
  • Every piece of the stack was new to me at once: NestJS, gRPC, Temporal, Pulsar, and the messaging backbone.
  • The outcome would decide whether the company's migration to the new stack was viable.
  • The message transport was fixed — the framework choice was the CTO's call; the data model and flows were mine to design.
  • The AI could not be trusted in the critical path without an exit: a stalled model must never leave a customer waiting.

Role and responsibility

I built the initial production version alone: the data model, the conversation flows, the agent-routing queue, the AI handoff path, and the operational patterns around them. As the system grew I brought in a second engineer, then handed the service off — first to a maintenance team, later to the AI team — and they run it independently today. The handoff was part of the job, not an afterthought: the service had to be operable by people who didn't build it.

Architecture

Buy the base, build the differentiators. I self-hosted an off-the-shelf messaging backbone as the message transport and built only what it doesn't have: an agent-routing queue that encodes the whole assignment policy — status first, load second, longest-wait tie-break — into a single Redis sorted-set score; a mini back-office view so agents see a member's account records, top-up / withdrawal history and permissions in one screen instead of switching systems; and an AI-first answer path with a 10-second race guard that hands the chat to a human whenever the model stalls or errors.

Routing queue

The score is banded so status always outranks load, and load outranks waiting time. Every agent state change is one atomic increment; assignment just reads the head of the queue — O(1), no scan, no race.

AI first, human takeover

AI answers first — designed to absorb the repetitive share of inquiries, roughly 70%. Every model call races a ten-second timer; on stall, error or timeout the conversation goes straight to a human via the routing queue. A per-room lock gives each conversation exactly one owner: the first invite wins, duplicates walk away.

Key decisions

Buy the base, build the differentiators

Options
Build the message transport from scratch, or self-host an off-the-shelf messaging backbone and build only what it doesn't have.
Decision
Self-host the backbone as transport (the framework was the CTO's call; the data model and flows were mine). Build the routing queue, the agent back-office view, and the AI handoff in-house.
Reason
Message delivery is a solved problem; agent routing, one-screen member context, and AI-with-takeover are where the product actually differs.
Trade-off
Operating third-party infrastructure we didn't write — its room and event model shaped how our flows had to be expressed, and its production behaviour was ours to learn.

Routing policy as one banded score

Options
Scan and rank agents on every assignment, or encode the whole policy into a single sortable number.
Decision
One Redis sorted-set score, banded so each term owns an order of magnitude: status first, load second, longest-wait tie-break. Assignment reads the head of the set.
Reason
Every agent state change becomes one atomic increment — no read-modify-write race — and picking the next agent is O(1).
Trade-off
The policy is compressed into an encoding: bands must never overflow into each other, and reading the queue means knowing the scheme. The banding later paid for itself when the Redis cluster rejected transactions — pure atomic increments were already enough.

AI answers first — behind a hard exit

Options
Use the model as a suggestion tool for agents, or let it answer customers directly with humans as the escalation path.
Decision
AI first, designed to absorb the repetitive share of inquiries — roughly 70% — with every model call raced against a ten-second timer and any stall, error or timeout handed straight to a human.
Reason
Most inquiries are repetitive; the ones that aren't need a person anyway. Treating the model as a flaky third-party dependency — timeout, fallback, never in the critical path without an exit — makes the design safe either way.
Trade-off
AI answering first means AI failing first: the escalation path has to be bulletproof, and the concurrency in that path — duplicate invites, racing score updates — turned out to be the hardest engineering on the project.

Failure modes and recovery

Real incidents and the mechanisms that came out of them.

The day the cluster said no

Queue positions lived as scores, and the original update was read-modify-write: read the score, compute in memory, write it back. Under concurrency two updates interleaved and one silently overwrote the other. I rebuilt the updates as atomic increments wrapped in a transaction — and then production revealed our Redis cluster wouldn't accept the transaction at all. Because the score design was banded — status, load and waiting time each own their own order of magnitude — every state change was already expressible as a single atomic increment, so we degraded to pure atomic ops the same day. A fix isn't done until it survives the real topology — and data designed for plan B is why the fallback was a same-day change, not a redesign.

Two agents, one conversation

Two messages arriving in the same conversation at the same moment each decided a human agent was needed — and two agents got invited into one chat. The fence is a per-room lock: the first caller takes it and performs the invite, everyone else finds it held and walks away, and the lock's time-to-live is aligned to the operation so a crash can't leave the room locked forever. Anything triggered by "a message arrived" will eventually be triggered twice in the same millisecond — design for the second caller before it exists.

The model is a dependency you can't trust

AI answering first means AI failing first. Every model call races a ten-second timer, and on stall, error or timeout the conversation is handed straight to a human — the user always ends up with a person, never with a spinner. The model gets treated like any flaky third-party dependency: timeout, fallback, and never in the critical path without an exit. That guard fired in production more than once, and nobody noticed — which was the point. The service's structured logging is what made that verifiable after the fact: the fallbacks are in the logs, not in the complaints.

Representative code

Rewritten for publication to show the production design — not company source.

routing-score.ts 
// One number encodes the whole routing policy — banded so each term
// owns an order of magnitude and can never outvote the one above it.
const score =
  statusLevel * 1e18 + // status ▸ away sinks below every online agent
  chatCount * 1e15 +   // load ▸ fewer active chats ranks first
  timestampMs;           // waited-longest ▸ idle-since breaks the tie

// Every state change is one atomic increment — no read-modify-write race.
await redis.zincrby(QUEUE, +1e18, agentId); // agent steps away
await redis.zincrby(QUEUE, -1e18, agentId); // agent comes back online

// Assignment is O(1): the best agent is simply the head of the set.
const [next] = await redis.zrange(QUEUE, 0, 0);
agent-invite.ts 
// Two messages in the same room, same millisecond, both want a human —
// without a fence, two agents get invited into one chat. First caller wins.
async function inviteAgent(roomId: string) {
  const locked = await redis.set(
    `agentRoomLock:${roomId}`, '1', { nx: true, px: 10_000 });
  if (!locked) return; // someone else already took the room — walk away

  const agent = await assignFromQueue(roomId);
  await notify(agent, roomId); // TTL aligned to op: a crash can't lock forever
}
ai-race-guard.ts 
// AI answering first means AI failing first. The model is a flaky
// dependency: race it against a timer, and on stall or error fall
// straight through to a human — the user never gets a spinner.
try {
  const reply = await Promise.race([
    askModel(message),
    rejectAfter(10_000), // ten seconds, then it's a person's job
  ]);
  return send(reply);
} catch {
  return forceToAgent(message); // timeout OR error → human takeover
}

Testing and validation

The service follows the platform's test discipline: Jest with integration tests running against real dependencies via Testcontainers — the lock and score behaviour is concurrency-dependent, so it's exercised against a real Redis, not a mock — behind an 80% coverage gate and a weekly regression run. The tests that matter most here fire the same trigger twice: two simultaneous invites into one room must produce exactly one assigned agent, and a model call that never returns must land the conversation on a human within the guard window.

Trade-offs

  • Self-hosting an off-the-shelf backbone meant operating infrastructure we didn't write — its room and event model constrained how our flows were expressed, in exchange for not rebuilding message delivery.
  • Compressing the routing policy into one banded score makes assignment O(1) and race-free, but couples the policy to an encoding: the bands are a contract, and changing the policy means changing the scheme.
  • AI answering first pushed real engineering cost into the escalation path — the race guard, the per-room lock, the atomic queue updates all exist because the fallback has to be more reliable than the model it backs.
  • Being the first service on the stack meant setting patterns while still learning it; some of that learning happened in production, which is where the war stories above come from.

Outcome

  • First production service on the company's new stack — the migration's viability was proven on it.
  • Its patterns — config via Vault, contract-first gRPC, structured logging, graceful shutdown — became the pattern reference later services copied.
  • AI-assisted handling designed to absorb roughly 70% of repetitive inquiries, with silent human takeover when the model stalls or errors.
  • Grew from a solo build to a two-person team, then a clean handoff — first to a maintenance team, later to the AI team, who run it independently.
  • Runs today as a standalone multi-channel middleware.

What I learned

Buy the commodity and build the differentiator — the transport was never where this project's value was. Anything triggered by "a message arrived" will eventually fire twice in the same millisecond, so conversation ownership has to be a fence, not a convention. A model in the critical path is a flaky dependency and gets a flaky dependency's treatment: timeout, fallback, exit. And a fix isn't done until it survives the real production topology — the banded score design was worth more on the day the cluster rejected transactions than on the day it was drawn. Finally, patterns set by a first service outlive the service: the most durable output here wasn't the support system, it was the template everything after it copied.
