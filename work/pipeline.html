Cross-Stack Rewards Pipeline

Four services, three languages, a reward path where a double-pay is real loss — the layer that makes payout happen exactly once, with zero legacy changes.

Role

Designed the exactly-once reliability layer — the owned ledger and idempotency key. The service decomposition follows the platform's existing seams and wasn't mine to claim.

Scale

Four services in three languages; real-time detection on the event stream; one durable 7-day workflow per enrolled user.

Stack

Java/Flink stream job · Rust + Temporal workflow engine · NestJS reward facade · atomic claims ledger.

Outcome

Exactly-once under both workflow replays and message redeliveries; zero changes to legacy services; verified end-to-end in pre-production.

Problem

Marketing wanted automatic, time-boxed missions for users close to their next loyalty tier: detect them in real time, run a 7-day window, and grant the tier upgrade instantly on success. That grant sits on a financially sensitive path — a duplicate payout is a real loss, not a cosmetic bug.

The hard part is that duplicates arrive from two independent sources. The event stream delivers at-least-once, and the workflow engine replays histories on recovery — so the grant step can legitimately run twice even when every component is behaving correctly. The pipeline also spans four services in three languages, and the legacy reward systems it ends in could not be modified.

Constraints

  • Financial correctness — a duplicate grant is a real loss; the payout must happen exactly once per user, tier, and month.
  • Two independent duplicate sources — at-least-once delivery from the stream and history replay from the workflow engine — both had to be neutralized by the same design.
  • Zero changes to legacy services — the grant had to go through the existing upgrade path untouched.
  • Cross-stack composition — four services in three languages, so the correctness guarantee could not rely on any one runtime's features.
  • Config consistency — mission thresholds used by more than one service could never be allowed to disagree.

Role and responsibility

My design is the reliability layer on top of the pipeline: the atomic claims ledger that gates enrollment, the human-readable idempotency key that gates the grant, and the single-owner rule for thresholds. I built the chain end-to-end across the four services, composing the platform's proven engines rather than building new ones, and I put the rollout through a dry-run-by-default plan and an outside red-team before calling it safe. The decomposition into detect / orchestrate / grant follows boundaries the platform already had — I claim the layer that makes it correct, not the seams.

Architecture

Three stages on existing platform seams: a Java/Flink job filters the event stream and computes entry-anchored goals from a lookup table; a Rust + Temporal engine runs one durable workflow per user for the 7-day window; a NestJS facade grants the upgrade through the existing legacy path. Exactly-once is layered on top — an atomic claims ledger (one chance per user, tier, month) gates enrollment, and a human-readable idempotency key gates the grant.

Detect → orchestrate → grant

Each stage reuses an engine the platform already runs: the streaming platform for detection, the workflow engine for the time-boxed window, the reward facade for the grant. Nothing new was built where something proven existed.

Walkthrough — how a duplicate dies

Enrollment: the ledger accepts exactly one claim per user, tier, and month — a redelivered detection event collides and stops. Grant: the key mission:user:tier:month is inserted before the payout; a workflow replay finds the key already present and returns without paying. Either way: the user is upgraded once, and the key is readable at a glance when someone has to debug it.

Key decisions

Compose existing engines, not build new ones

Options
Build a bespoke mission service owning detection, timing and payout — or place each stage on an engine the platform already runs.
Decision
Boundaries follow existing seams: the streaming platform detects, the workflow engine holds the 7-day window, the reward facade grants.
Reason
Every engine in the chain was already proven in this platform; the only genuinely new problem was making the composition exactly-once.
Trade-off
Four services in three languages means no single runtime can carry the correctness guarantee — it had to live in shared data, not in any one framework's features.

Layered exactly-once with a human-readable key

Options
One dedupe check at the payout, an opaque hash key, or two independent gates — ledger at enrollment, idempotency key at the grant.
Decision
Both gates, and the key is readable: mission:user:tier:month.
Reason
Duplicates come from two independent sources — message redelivery and workflow replay — so a single gate leaves one source uncovered. A readable key means an on-call engineer can see at a glance who was granted what, and why a second attempt was refused.
Trade-off
Two writes on the happy path instead of one, and the key format becomes a contract — changing it later would invalidate the dedupe history.

Zero legacy changes — grant through the existing path

Options
Modify the legacy reward systems to understand missions, or wrap the existing upgrade path behind a thin facade and leave legacy untouched.
Decision
A NestJS facade calls the existing upgrade path as-is; the mission concept never leaks into legacy code.
Reason
The legacy path already carries the platform's real upgrades — reusing it means the grant behaves identically to every other upgrade, with no new code on the sensitive side.
Trade-off
The pipeline inherits the legacy path's behavior and limits wholesale; anything it can't express, the mission can't grant.

Failure modes and recovery

The failure modes this layer was designed against — each one is structural to the platform, not hypothetical.

The message that arrives twice

The event stream is at-least-once by contract: the same detection event can be redelivered after a consumer restart or a rebalance. Without a gate, a redelivered event enrolls the same user twice and the mission pays twice. The ledger makes enrollment an atomic insert — one chance per user, tier, and month — so the second delivery collides on the claim and stops. Recovery is nothing: the duplicate simply dies at the door.

The workflow that runs its history again

The workflow engine recovers by replaying history on another worker — which means the grant step can execute again even though the first attempt succeeded. The idempotency key is inserted before the payout: a replay finds mission:user:tier:month already present and returns without paying. The two gates are independent on purpose — either source of duplicates is stopped even if the other gate were bypassed.

Two repos, two versions of the same number

Mission thresholds are consumed by more than one service. If each repository carried its own copy, a config edit in one place would silently split the pipeline's view of the goal. So every threshold has exactly one owner — a lookup table — and producers broadcast unconditionally; consumers never hold a local copy that can drift. Two repositories can never disagree about the same number, because only one of them has it.

The goal that changes mid-mission

A 7-day window outlives config edits. If a threshold is raised on day three, users already running would see their goal move under them. Goals are entry-anchored: computed once at enrollment from the lookup table and fixed for the life of the workflow — later edits apply to new enrollments only. A running mission always finishes under the rules it started with.

Representative code

Rewritten for publication to show the production design — not company source.

exactly-once.ts 
// Two independent sources of duplicates — message re-delivery AND workflow
// replay. One atomic ledger claim per user·tier·month neutralizes both.
async function grantOnce(userId: string, tier: number, month: string) {
  const key = `mission:${userId}:${tier}:${month}`; // human-readable, debuggable

  // Insert-if-absent is the whole fence: the second caller collides and stops.
  const claim = await ledger.insertUnique({ _id: key, at: Date.now() });
  if (!claim.inserted) return; // already granted — a replay, not a new reward

  await legacyUpgradePath.grant(userId, tier); // reuse old system, 0 changes
}
thresholds.ts 
// Every threshold has exactly one owner — a lookup table. Producers
// broadcast unconditionally, so two repos can never disagree on a number.
const GOALS: Record<number, number> = { 1: 500, 2: 2_000, 3: 10_000 };

// Entry-anchored: the goal is fixed at enrollment, immune to later edits.
function goalFor(tier: number): number {
  const goal = GOALS[tier];
  if (goal === undefined) throw new Error(`no goal for tier ${tier}`);
  return goal;
}

Testing and validation

The pipeline follows the platform's standard test discipline — Jest with integration tests against real dependencies via Testcontainers, and every path covered by the weekly regression suite. The tests that matter most here mirror the two duplicate sources: deliver the same detection event twice and assert one enrollment; drive the grant step through a replay and assert one payout.

Beyond tests, the chain was verified end-to-end against production-like traffic in the pre-production lane, with funnel metrics at every stage — detected, enrolled, completed, granted — so a lost or doubled record is visible as a stage-to-stage mismatch. The rollout itself was dry-run by default, and I put the plan through an outside red-team before calling it safe.

Trade-offs

  • Layered exactly-once costs an extra write and an extra collection on the hot path — accepted, because either duplicate source alone can produce a real loss.
  • The human-readable key is a contract: its format can't change without invalidating the dedupe history, so it's deliberately boring and minimal.
  • Granting through the untouched legacy path means inheriting its behavior and limits wholesale — the price of zero legacy changes.
  • Three languages across four services keeps each stage on its best engine, but every cross-service guarantee has to live in data, and debugging spans three toolchains.

Outcome

  • Exactly-once payout under both workflow replays and message redeliveries — the two duplicate sources are gated independently.
  • Zero changes to legacy services; the grant reuses the existing upgrade path as-is.
  • Verified end-to-end against production-like traffic in the pre-production lane (QAT) — honest scope: pre-production, not production.
  • Funnel metrics at every stage and a dry-run-by-default rollout, red-teamed before being called safe.
  • The layer — owned ledger, readable key, detect / orchestrate / grant with single-owner config — is the template for the next mission types.

What I learned

When a pipeline crosses runtimes, correctness can't be borrowed from any one framework — it has to live in data both sides can see, which is why the ledger and the key carry the guarantee instead of any engine feature. Duplicates from independent sources need independent gates; one check that "should" cover both is one bug away from covering neither. And observability is part of the design, not an add-on: a key a human can read and a funnel a human can diff are what make an exactly-once claim checkable rather than taken on faith.
