AI Delivery Pipeline

A staged delivery pipeline where AI drafts and humans gate every merge — built for a small team with a growing backlog.

Role

Designed and built the pipeline, its skills and its review agents; my team ships through it.

Scale

10 pipeline stages · 23 skills · 12 role agents · a routing manifest across 17 repositories.

Stack

Agent skills (Markdown policies) · MCP · proto-first contracts · content-hash artifact state.

Outcome

16 features shipped through it; feature first drafts went from days to hours. Nothing merges without human sign-off.

Problem

A small backend team, a backlog that grows faster than headcount, and AI tools that produce plausible-but-wrong output when used casually. Unmanaged agent use had real costs: an agent "fixed" a harmless legacy bug that other code depended on and broke production behaviour; another answered a one-condition change with hundreds of new lines; another updated a database row by row — correct, and an IO storm at our volume.

The pattern behind all of these: damage happens when someone can't yet tell a good answer from a plausible one. The problem to solve was not "use AI more" — it was making AI output safe to ship.

Constraints

  • Nothing irreversible without a human: commit, publish, deploy, migration all require explicit confirmation.
  • Contract discipline — producers and consumers must not drift, so contracts are generated and shared before code is written.
  • Finite context — a model can't hold 17 repositories; relevance had to be engineered, not hoped for.
  • Auditability — every stage's output is a reviewable artifact with an approval signature.
  • Junior safety — the pipeline had to protect people still building the discernment to judge AI output.

Role and responsibility

I designed the pipeline, wrote the stage skills and agent remits, and set the governance rules the team runs under. Some building blocks were adopted from the community and adapted — credited, not claimed. The delivery pipeline, the review roles and the repo router were built from scratch. The team uses it daily; I maintain it.

Architecture

A one-line feature request walks through ten artifact-gated stages: requirements draft → design proposal → human design approval → contract generation → SDK update → task breakdown → parallel implementation by subagents → machine-judged verification against the contracts. There is no state database — every stage derives its state from the presence and content hash of the artifact the previous stage wrote, and refuses to run until that artifact exists and is approved.

What AI drafts: requirements documents (unknowns marked "to confirm" instead of guessed), design proposals, contracts, task breakdowns, implementation, review feedback. What humans own: the design approval gate, every merge, and every irreversible action. What can never happen automatically: commit, publish, deploy, migration.

Key decisions

Artifact-gated state, no state database

Options
Track pipeline progress in a state store, or derive state from the artifacts themselves.
Decision
Content-hash state: each stage hashes the upstream artifact and detects changes.
Reason
Edits ripple forward deterministically; re-running a stage is always safe, and the artifact trail doubles as the audit trail.
Trade-off
Ceremony — even a small change walks the stages, which can feel heavy for one-line fixes.

Specialist reviewers, not one generalist agent

Options
One agent that does everything, or twelve role agents with tight remits and separate tool budgets.
Decision
Role specialists — planner, architect, developers, reviewers, QA, security auditor and more. None can approve its own work.
Reason
Each role draws a boundary; review quality comes from the separation, the same way it does in a human team.
Trade-off
More invocations and more harness maintenance than a single-agent setup.

Contracts before code

Options
Let implementation drive the interfaces, or generate and publish contracts before a line of code is written.
Decision
Contract-first: the design stage produces the API and data contracts; the SDK consumes them before development starts.
Reason
Producers and consumers can't drift, and machine-judged acceptance has something objective to judge against.
Trade-off
Upfront design cost on features where the contract is trivial.

Failure modes and recovery

The discipline came from the failures — these are the ones that shaped the rules.

The fix that broke production

The agent found a real bug — one that had sat harmless in production for years — and repaired it. The repair changed behaviour other code had quietly grown to depend on, and things blew up. An agent evaluates code; it doesn't know your system's history. A harmless bug is load-bearing until proven otherwise.

The big answer to a small ask

The requirement was genuinely small — an existing rule needed one more condition, a change the codebase was already shaped for. The agent, not grasping that intent, rebuilt the surrounding flow instead: hundreds of new lines that technically worked and doubled the review burden. The smallest correct change comes from knowing what the codebase already wants to do — that judgment is still ours.

Correct, and slow

Asked for accuracy, the agent updated the database row by row — correct, and an IO storm at our volume. Batching was the obvious answer, but only obvious to someone who has paid for IO before. Performance instincts don't ship with the model; they come from the scars.

How failures become policy

Each of these compiled into the pipeline: a production lesson becomes a rule wired into the agent harness — a readiness gate the agent can't skip — so the same mistake can't ship twice. The rule is not "use AI less." It is: discernment first, delegation second.

Representative artifact

Real excerpt from the pipeline repo — desensitized (neutral service names, no company internals). Each stage and building block is a skill: a Markdown policy the agent reads before it acts. This one came out of a real incident.

skills/graceful-shutdown/SKILL.md 
# Graceful Shutdown — Monorepo Default Policy

# A service is "graceful-restart capable" iff all six hold:

  1. Signal binding. Listen for SIGTERM → enter draining state, do not exit.
  2. Two-stage drain. Flip health probe to NOT_SERVING first, wait one
     readiness period so the load balancer stops sending traffic, then drain.
  3. In-flight wait. Stop accepting new connections; let in-flight
     requests finish, bounded by a timeout < terminationGracePeriodSeconds.
  4. Background workers drained. Consumers, tickers, span processors
     all observe the shutdown signal and exit cleanly.
  5. Connection cleanup. DB / Redis / upstream pools closed after the
     server stops; telemetry flushed last.
  6. K8s manifest aligned. terminationGracePeriodSeconds ≥ drain + buffer.

# If verification can't be done, say so in the PR —
# do not claim "graceful shutdown done" on code review alone.

Testing and validation

The final stage is machine-judged acceptance against the contracts generated at design time — objective criteria, not vibes. Code produced through the pipeline still passes the same bar as any other code on the platform: review by the role agents and by humans, the coverage gate, and the weekly full regression before release. Every artifact carries a human approval signature before the next stage will run.

Trade-offs

  • Review burden is real: AI multiplies draft volume, and every line still needs a human who can judge it.
  • Hallucination risk never reaches zero — the gates reduce blast radius; they don't remove the need for discernment.
  • Context quality is a standing maintenance cost: the routing manifest and 30+ per-service context files have to stay current or the agent starts every task wrong.
  • Skills and agent remits are code too — 23 skills need versioning, review and upkeep.
  • False confidence is the quiet failure mode: output that looks right invites rubber-stamping, which is why no agent can approve its own work.

Outcome

  • 16 features shipped through the pipeline.
  • Feature first drafts compressed from days to hours — a two-day build becomes a two-hour first draft; the last twenty percent is still ours.
  • No unreviewed merge: every stage produces a signed-off artifact before the next runs.
  • Production lessons compile into pipeline policy, so the same mistake can't ship twice.

What I learned

AI amplifies your code culture — an agent raised on a disciplined codebase ships discipline; raised on mess, it ships mess faster. Context beats prompts: the agent should start every task already knowing the codebase, not be talked into it per request. And the human skill that matters most is discernment — you must be able to judge the problem and the logic before you outsource the typing.
