Scaling the Platform

Improved core platform mechanisms as daily active users grew from 50k to 400k+.

Role

Drove the scaling work as the platform grew — measured the bottlenecks, designed the replacements, and landed them with the team inside the weekly release train.

Scale

50k → 400k+ daily active users over four years; every change shipped live, with users on.

Stack

Redis (INCR · Pub/Sub · SSCAN) · MongoDB · WebSocket gateway · TypeScript services.

Outcome

Old failure modes gone at 400k+ DAU; sub-second pushes; 30-day report queries where one day used to time out; every cutover zero-downtime and revertible.

Problem

Over four years the member management platform grew from 50k to 400k+ daily active users, and growth kept finding the limits of the old mechanisms. A counter document in the database serialized every ID issued. Notification fan-out ran from one process to every connection and stalled under load. Blocking full-set scans held Redis while they ran. An in-memory config cache grew without bound and leaked. Report queries on the busiest collection couldn't return a single day's records without timing out.

More than once, the failure happened in production first. Each of these was a live global single-point — the question was never just what to replace it with, but how to replace it with users online.

Constraints

  • No downtime allowance — the platform stayed live through every change, users on.
  • No rewrite budget — every fix had to land inside the weekly release train, alongside normal feature work.
  • Every target was a live global single-point, so each step needed its own rollback before it shipped.
  • ID generation could never produce a collision — old and new generators had to be safe to run at the same time.

Role and responsibility

I drove this work as the platform grew: I championed bringing a distributed cache into the platform, measured where the time actually went before proposing changes, root-caused each bottleneck to its mechanism, and designed the replacements and their cutover plans. Implementation and delivery were shared with the team — the changes shipped through the same weekly release train as everything else.

Architecture

The principle was to replace the mechanism, not the symptom. ID generation moved from a database counter document to atomic Redis INCR. Notification fan-out moved to Redis Pub/Sub: publish once per event, and channel names route each user's events so an instance only touches the sockets it actually holds. Blocking SMEMBERS scans became cursor-based SSCAN. The leaking in-memory config cache was rebuilt cache-aside — read local, fall through to Redis, refresh on a fixed 15-minute cycle. Underneath it all, the query layer was retuned: covering indexes matched to real query shapes, unused indexes removed, aggregation pipelines rebuilt, text search moved off regex scans onto a proper search index.

Before → after

Database counter → Redis INCR with a +1-billion offset. Single-process WebSocket fan-out → Pub/Sub across N instances. Blocking SMEMBERS → SSCAN cursors. Unbounded in-memory cache → bounded cache-aside with a fixed refresh cycle.

Walkthrough — how a live cutover ran

Seed: the new mechanism starts in a range or path that cannot collide with the old one — for IDs, one billion above the old counter. Run both: old and new operate at the same time; neither can corrupt the other. Cut over: traffic moves in small, individually revertible steps, one release at a time. Rollback: boring on purpose — point back at the old mechanism. Each step's rollback existed before the step shipped.

Key decisions

ID cutover by offset, not choreography

Options
Choreograph a precise handover between two live ID generators, or start the new range one billion above the old counter so both can run at once.
Decision
The +1-billion offset. Old and new generators ran in parallel and could never collide.
Reason
With users online and no maintenance window, the safest cutover is the one where collision is impossible by construction, not prevented by timing.
Trade-off
A permanent billion-wide gap in the ID sequence, and ID issuance now depends on Redis being up — a new availability dependency the team had to operate.

Move fan-out to Pub/Sub instead of tuning the sockets

Options
Tune the WebSocket layer — the visible component — or measure first and change whatever the profile pointed at.
Decision
Measurement showed the cost was the single-process fan-out loop, not the sockets. We published each event once to Redis Pub/Sub; every instance subscribes and delivers only to the connections it holds.
Reason
Channel-per-user routing removes the single fan-out point entirely and scales delivery with the number of instances.
Trade-off
Redis Pub/Sub is fire-and-forget — an instance that is down during publish misses the message — so delivery behaviour under Redis failure had to be understood and accepted for notifications.

Unit of change = unit of rollback

Options
Ship each migration as one big change, or as a series of small, individually revertible steps.
Decision
Small steps in series, each with its own rollback. The fan-out migration taught us this the hard way: the first landing shipped as one change, production surprised us, and we reverted the whole thing.
Reason
On a live platform the design being right isn't enough — the delivery has to be revertible at the same granularity it ships.
Trade-off
Slower delivery: more releases per migration, and a longer window where old and new mechanisms run in parallel and both need watching.

Failure modes and recovery

What broke, and the practices that came out of it.

Reports that stopped coming back

One stretch of growth broke the reporting layer visibly: operational reports stopped returning, and on the busiest collection a query couldn't fetch a single day's records without timing out. Recovery was months of unglamorous work — reading the slow-query log, matching covering indexes to the query shapes people actually ran, deleting the indexes nobody used (on write-heavy collections an unused index is pure tax), rebuilding aggregation pipelines stage by stage, and moving text search off regex scans onto a search index. Thirty-day report windows came back comfortably where one day used to time out.

Measure first, then argue

Push notifications were arriving late, and the room's instinct was to tune the WebSocket layer — the visible component. Measuring where the time actually went showed the cost was the fan-out loop behind the sockets, delivering to every connection from one place. Fan-out moved to Pub/Sub and pushes went sub-second. The loudest component is rarely the bottleneck; the profile decides.

The revert that came back stronger

The fan-out migration shipped — and production surprised us, so we reverted. The second landing was a series of small, individually revertible steps, each hardened by what the first attempt taught us. Nothing about the design was wrong; the delivery was. The rule that came out of it — the unit of change has to equal the unit of rollback — applied to every migration after.

Representative code

Rewritten for publication to show the production design — not company source.

id-cutover.ts 
// Live cutover with users online: the new range starts one billion above
// the old counter, so both generators can run at once and never collide.
const OFFSET = 1_000_000_000;

const legacy = await db.counters.findOne({ _id: 'memberId' }); // old ceiling
await redis.set('seq:memberId', legacy.seq + OFFSET, { nx: true }); // seed once

const nextId = await redis.incr('seq:memberId'); // atomic — no document lock
// Rollback is boring on purpose: point back at the old counter.
push-fanout.ts 
// The bottleneck was never the sockets — it was one process fanning out
// to every connection. Publish once; each instance delivers to its own.
export function broadcast(userId: string, event: Event) {
  return redis.publish(`push:${userId}`, JSON.stringify(event));
}

// Every instance subscribes and only touches sockets it actually holds.
redis.psubscribe('push:*', (channel, payload) => {
  const userId = channel.slice('push:'.length);
  localSockets.get(userId)?.send(payload); // sub-second, N instances
});
config-cache.ts 
// The old in-memory config cache grew without bound and leaked. Rebuilt
// cache-aside: read local, fall through to Redis, refresh on a fixed cycle.
async function getConfig(key: string): Promise<Config> {
  const hit = local.get(key);
  if (hit && hit.expires > Date.now()) return hit.value;

  const value = await redis.get(key).then(JSON.parse);
  local.set(key, { value, expires: Date.now() + 15 * 60_000 });
  return value; // bounded, self-refreshing — config edits go live in minutes
}

Testing and validation

The platform-wide practice applied here: Jest with Testcontainers against real databases rather than mocks, an 80% coverage gate, and a full regression pass every week before release — which is what made shipping migrations inside the release train tolerable. Beyond the suite, every mechanism change was validated by measurement: profile before to find the real bottleneck, run old and new in parallel where possible, and watch the same metric after each revertible step before taking the next one.

Trade-offs

  • Operational complexity: Redis went from a cache to core infrastructure — ID issuance, notification delivery and config reads all depend on it, so its memory, monitoring and failover became things the team had to own.
  • Redis failure behaviour: Pub/Sub is fire-and-forget, so an instance that is down at publish time misses the message — acceptable for notifications, and part of why the audited transaction path never moved onto it. If Redis is unavailable, new ID issuance stops rather than risking a collision.
  • Observability requirements: revertible steps only work if you can see them working — each cutover needed before/after measurement and a watch window, and during migration both old and new mechanisms had to be monitored at once.
  • Consistency: the cache-aside config layer trades freshness for boundedness — an edit can take up to the 15-minute refresh cycle to converge across instances, a window that had to be acceptable to the people changing configs.
  • Migration in small steps was slower than a big-bang cutover would have been — more releases, and a longer parallel-run tail.

Outcome

  • The platform sustains 400k+ daily active users without the old failure modes.
  • Push notifications went from visibly late to sub-second, delivered across N instances.
  • Thirty-day report queries return comfortably on collections where a single day used to time out.
  • ID generation cut over live with zero downtime and zero collisions; rollback stayed one config change away throughout.
  • The patterns introduced — cache-aside, atomic counters, pub/sub fan-out — became house standards for modules built after.

What I learned

Measure before arguing — the visible component is rarely the bottleneck, and the profile settles debates that intuition keeps open. Replace mechanisms, not symptoms: tuning a serialized counter or a single-process fan-out loop only postpones the same failure. And on a live platform, delivery design matters as much as system design — the +1-billion offset and the reverted-then-relanded fan-out both taught the same lesson: prefer the change with nothing left to go wrong, and make its unit of change equal its unit of rollback.
